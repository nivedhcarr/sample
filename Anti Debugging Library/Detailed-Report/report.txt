
Creating an anti-debugging library is a classic high-level project for a reverse engineer because it requires you to understand the exact artifacts and behaviors that debuggers leave behind.
To make it truly "advanced," your library should go beyond simple Windows API calls like IsDebuggerPresent() and instead use low-level system interactions that are harder for analysts to hook or patch. 
Core Anti-Debugging Categories to Implement
You can structure your library into several "detection modules" based on how they find the debugger.

    Environmental Artifact Checks: Search for "telltale signs" of a debugger's presence.
        PEB Flags: Directly read the Process Environment Block (PEB) to check flags like BeingDebugged or NtGlobalFlag without using official APIs.
        Registry & Files: Look for registry keys or files associated with tools like x64dbg, IDA Pro, or Wireshark.
    Exception-Based Detection: Leverage how debuggers handle errors differently than the OS.
        Breakpoint Detection: Scan your own code in memory for 0xCC (INT 3) instructions, which indicate software breakpoints.
        Structured Exception Handling (SEH): Intentionally trigger exceptions (like CloseHandle with an invalid handle) and check if a debugger "swallows" the error before your own SEH handler can catch it.
    Hardware-Level Checks: Monitor CPU-specific features used by advanced analysts.
        Debug Registers (DR0-DR3): Use RtlCaptureContext() to check if hardware breakpoints are set in the CPU's debug registers.
        Timing Attacks: Measure the time delta between two points in your code using RDTSC. If a debugger is stepping through the code, the time difference will be massive compared to normal execution.
    System Call Evasion: Instead of calling ntdll.dll functions that analysts often hook, use Direct Syscalls. This allows your library to talk directly to the kernel, bypassing user-mode monitoring tools. 

Advanced Implementation Ideas

    Self-Modifying Code: Have your library decrypt its own detection logic at runtime so that static analysis can't find the "kill switch" code.
    Thread Hiding: Use undocumented flags like ThreadHideFromDebugger (0x11) via NtSetInformationThread to prevent a debugger from even seeing your protection thread.
    Cross-Platform Support: Build modules for both Windows (PEB, SEH) and Linux (ptrace checks, /proc/self/status inspection) to understand how different kernels manage process tracing.

where to start from?
The best place to start is with the Process Environment Block (PEB). It’s the "ground zero" of anti-debugging because it’s a data structure maintained by the OS for every process, and it contains several flags that change the moment a debugger attaches.
Starting here teaches you how to move away from high-level APIs (which are easily hooked) and toward direct memory manipulation.
Phase 1: The "Manual" IsDebuggerPresent
Instead of calling IsDebuggerPresent(), which just returns a boolean, write a C++ function that manually crawls the PEB.

    Access the PEB: On x64 Windows, the GS register points to the Thread Information Block (TIB). The PEB is located at an offset from there.
    Check the BeingDebugged flag: This is a single byte at offset 0x02 in the PEB.
    Check NtGlobalFlag: At offset 0xBC, this flag usually has a value of 0x70 (a combination of heap flags) if a debugger is present.

bool CheckPEB() {
    // Read the PEB pointer from the GS register
    unsigned long long peb = __readgsqword(0x60); 
    
    // Check the BeingDebugged flag at offset 0x02
    unsigned char beingDebugged = *(unsigned char*)(peb + 2);
    
    return beingDebugged != 0;
}

predefined?
It is a compiler intrinsic. 
In C++, an intrinsic is a "predefined" function built directly into the compiler rather than being part of a separate library. When the compiler sees __readgsqword, it doesn't "call" a function; it replaces that line with a single, highly efficient assembly instruction. 
What exactly does it do?
It reads memory relative to the GS segment register on 64-bit systems. In assembly, the code __readgsqword(0x60) becomes something like: 
nasm

mov rax, gs:[60h]

Use code with caution.
This tells the CPU: "Go to the memory address stored in the GS register, add 0x60 to it, and give me the 8-byte value (the Quad-Word) found there". 
Key Facts

    Availability: It is specific to the Microsoft C++ (MSVC) compiler and is only available when targeting 64-bit (x64) architectures.
    Header: To use it, you typically need to include <intrin.h>.
    Purpose: It provides a way to access hardware-specific features (like segment registers) that standard C++ cannot normally reach. 

In your specific example, 0x60 is the "magic number" (offset) where Windows stores the pointer to the Process Environment Block (PEB) within the thread's local data. 


unsigned char beingDebugged = *(unsigned char*)(peb + 2);
This line is pointer arithmetic used to "read" a specific byte from memory.
Here is the breakdown of why it's written that way:
1. (peb + 2)
We take the base address of the PEB (which we just got from the GS register) and add 2 bytes to it.

    In the Windows PEB structure, the first two bytes are different flags.
    The third byte (at offset 0x02) is the BeingDebugged flag.

2. (unsigned char*)
This is a type cast.

    The variable peb is just a number (an address).
    By casting it to an unsigned char*, we are telling the compiler: "Treat the memory address at peb + 2 as the location of a single byte (8 bits)."

3. The * (Dereference)
The asterisk at the very beginning of *(unsigned char*)... is the dereference operator.

    It tells the CPU: "Don't just give me the address; go into that address and bring back the value stored there."

Putting it all together:

    Start at the PEB address.
    Move 2 bytes forward.
    Look at that specific spot.
    Grab the 1-byte value found there.
    Save that value into the variable beingDebugged.

If a debugger is attached, the Windows Kernel automatically sets this specific byte to 1. If not, it stays 0.


return beingDebugged != 0;
what's this
This is the final verdict of the function. It converts the byte value we just read into a simple true or false answer.
Here is the logic breakdown:
1. The Comparison (!= 0)
The variable beingDebugged is a number (a byte). In the Windows PEB:

    0 means "No debugger found."
    1 (or any non-zero value) means "A debugger is attached."

The code asks: "Is the value we found NOT equal to zero?"
2. The Boolean Return
Because the function is defined as bool CheckPEB(), it must return either true or false.

    If beingDebugged is 1, the statement 1 != 0 is True. The function returns true.
    If beingDebugged is 0, the statement 0 != 0 is False. The function returns false.

3. Why write it this way?
In C and C++, you could technically just write return beingDebugged;, because any non-zero number is treated as "true." However, writing != 0 is considered cleaner coding practice because:

    It explicitly forces the result into a bool (0 or 1).
    It makes the programmer's intent crystal clear to anyone reading the code later.

1. The Logic
In the PEB (which you accessed in Phase 1), there is a field called ProcessHeap. This is a pointer to the default heap of the process. Inside that heap structure, two fields change significantly under a debugger:

    Flags: Usually HEAP_GROWABLE (0x2). If a debugger is present, other flags like HEAP_TAIL_CHECKING_ENABLED are added.
    ForceFlags: Usually 0. If a debugger is present, this becomes non-zero.




#include <iostream>
#include <intrin.h> // For __readgsqword

bool CheckHeapFlags() {
    // 1. Get PEB address from GS:[0x60]
    unsigned long long peb = __readgsqword(0x60);

    // 2. Get ProcessHeap pointer at offset 0x30 in PEB
    unsigned long long processHeap = *(unsigned long long*)(peb + 0x30);

    // 3. Read Flags and ForceFlags
    // Note: Offsets vary slightly by Windows version, but for Win10/11 x64:
    // Flags is at offset 0x70, ForceFlags is at 0x74
    unsigned int flags = *(unsigned int*)(processHeap + 0x70);
    unsigned int forceFlags = *(unsigned int*)(processHeap + 0x74);

    // 4. Typical detection logic
    // Normal: Flags == 0x2, ForceFlags == 0
    if (forceFlags != 0) return true;
    if (flags != 0x2) return true;

    return false;
}

 Why this is "Advanced"
A standard analyst using IsDebuggerPresent() can just flip a single bit in memory to bypass your check. However, to bypass Heap Flag checks, the analyst must:

    Start the process without a debugger and attach later.
    Or, use a "Stealth" plugin (like ScyllaHide) that manually patches the heap structure of the target process.



1. The GS Register (0x60)
On x64 Windows, the GS segment register points to the Thread Information Block (TIB).

    At offset 0x60 of the TIB, Windows stores the address of the PEB (Process Environment Block).
    By using __readgsqword(0x60), you are asking the CPU for the "Master Map" of your process.

2. The ProcessHeap Pointer (0x30)
Inside that "Master Map" (the PEB), there are dozens of variables.

    At offset 0x30, there is an 8-byte pointer that tells the OS where your Default Process Heap lives in RAM.
    *(unsigned long long*)(peb + 0x30) follows that pointer to the actual Heap structure.

3. The "Gotcha" Offsets (0x70 and 0x74)
Once you are looking at the Heap structure:

    0x70 (Flags): This is a 4-byte bitmask. If a debugger is present, the OS adds flags like HEAP_VALIDATE_PARAMETERS_ENABLED.
    0x74 (ForceFlags): This is the smoking gun. It’s almost always 0 in a normal process. If it's anything else, a debugger likely launched the process.

💡 Pro-Tip for your Library: The "Version Trap"
While these offsets (0x70, 0x74) work for Windows 10 and 11, they occasionally change in older versions of Windows (like Windows 7).





should we combine it with heap tail
Yes, combining the flag checks with Heap Tail Checking makes your library significantly more robust.
While an analyst can easily patch a single if statement or flip a bit in the PEB, it is much harder for them to "fake" the actual physical structure of memory buffers in the heap.
Why combine them?

    Redundancy: If a stealth debugger (like ScyllaHide) successfully hides the Flags and ForceFlags in the PEB, it might still forget to disable the Heap Tail Fill mechanism.
    Physical Evidence: Flags are just numbers; the "Tail Fill" is actual data written to memory by the OS. It’s "smoking gun" evidence that the heap is in debug mode.

How it works
When a debugger is active, the Windows Heap Manager adds a "Tail Fill" (usually 8 bytes of 0xABABABAB) immediately after any memory allocation to catch buffer overflows.



#include <Windows.h>

bool CheckHeapTail() {
    // 1. Allocate a tiny bit of memory
    // In debug mode, the OS will add a "tail" to this allocation
    void* p = malloc(10);
    if (!p) return false;

    // 2. The tail starts immediately after the allocated size
    // We check the memory at p + offset
    // Note: The exact offset can vary by alignment, but usually it's right after.
    unsigned char* tail = (unsigned char*)p + 10;

    bool detected = false;

    // 3. Look for the magic pattern: 0xABABABAB
    // We check the first 4 bytes of the "tail"
    if (*(unsigned int*)tail == 0xABABABAB) {
        detected = true; 
    }

    free(p);
    return detected;
}






The "Advanced" Twist
To make this even harder to detect, don't use 0xABABABAB directly in your code (as analysts search for that constant). Instead, use a bitwise operation:
if (*(unsigned int*)tail == (0xAAAAAAAA ^ 0x01010101))





PHASE 3 

Now for the "predator" part of your library: Phase 3 – Timing Attacks.
While Phases 1 and 2 check for static artifacts (flags left in memory), Phase 3 detects the analyst's behavior. If an analyst is using a debugger to "step" through your code (F8/F10), they are thousands of times slower than a CPU. We use the RDTSC (Read Time-Step Counter) instruction to catch this delay.
1. The Concept
The CPU has a counter that increments every single clock cycle since the computer started.

    Normal Execution: Two lines of code might take 50–200 cycles.
    Debugging/Stepping: Between those same two lines, the analyst takes seconds to click "Step." The cycle count jumps by millions.

We use the __rdtsc() intrinsic to grab the "before" and "after" snapshots of the CPU clock.



Why this is "Advanced"

    Bypassing is hard: An analyst can't just flip a flag in the PEB. They have to manually patch your code to "skip" the timing check or use a kernel-mode driver to "spoof" the RDTSC value (which is very difficult).
    The "Junk Code" Trap: Notice the for loop in step 2. If you just do t1 = rdtsc(); t2 = rdtsc();, an analyst can easily see the two instructions next to each other. By putting junk code or legitimate logic in between, you hide the trap

#include <intrin.h>

bool CheckTiming() {
    unsigned long long t1, t2;

    // 1. Get initial timestamp
    t1 = __rdtsc();

    // 2. Execute a small, "noisy" block of code 
    // This makes it harder for simple automated bypasses
    for (int i = 0; i < 100; i++) {
        GetTickCount(); 
    }

    // 3. Get final timestamp
    t2 = __rdtsc();

    // 4. Threshold check
    // On a modern CPU, this loop should take < 50,000 cycles.
    // If it takes > 1,000,000, someone is likely stepping through it.
    if ((t2 - t1) > 0x100000) {
        return true; // Debugger detected!
    }

    return false;
}

That loop is "bait" designed to waste a tiny, predictable amount of time.
What is GetTickCount()?
It’s a standard Windows function that returns the number of milliseconds that have elapsed since the system was started.
In this specific anti-debug trick, we aren't actually using the result of GetTickCount(). We are just using the function as a speed bump.
Why use it in the loop?

    To Create a Gap: If you call __rdtsc() twice in a row with nothing in between, the time difference is almost zero. By running a loop of 100 API calls, you create a "normal" baseline of CPU cycles (e.g., 50,000 cycles).
    To Trigger the Analyst: When an analyst sees a loop calling a Windows API, they often "step over" it or pause to see what it's doing. The moment they pause or click "Step," the cycle count explodes from 50,000 to 5,000,000+, triggering your trap.
    To Look "Normal": Anti-debugging code is most effective when it looks like regular program logic. GetTickCount() is used in thousands of legitimate apps for timing and logging, so it doesn't immediately scream "MALWARE PROTECTION" to someone glancing at the code.

The Math

    Normal: Loop starts -> CPU does 100 quick tasks -> Loop ends. (Time: Short)
    Debugged: Loop starts -> Analyst clicks "Step" -> Analyst looks at registers -> Analyst clicks "Step" again. (Time: Enormous)

The Concept: The "Invalid Handle" Trap
Normally, if you try to close an invalid or fake handle using CloseHandle(), Windows just returns an error code. However, if a debugger is attached, Windows throws a CONTROL_C_EXIT or a BRP_INVALID_HANDLE exception.
The debugger will try to "catch" this crash to show it to the analyst. Your goal is to see if your code's error handler gets to handle it first, or if the debugger "swallows" it.

The Interrupt: When you call CloseHandle with a garbage value while debugging, the debugger will usually pause the program and say: "Exception Triggered: Invalid Handle."
The Reveal: If the analyst tells the debugger to "ignore" or "continue," the execution flow inside your __try/__except block is often altered.
The INT 3 Trick: Another way to do this is to manually insert an __debugbreak() (which is an INT 3 instruction).

    No Debugger: The program hits the __except block immediately.
    Debugger: The debugger catches the INT 3 and stops execution entirely, waiting for the analyst.

Phase 5: Thread Hiding (ThreadHideFromDebugger)
This is the most aggressive technique in your library. Instead of just checking for a debugger, you are actively attacking the debugger’s ability to see your code.
1. The Concept
In Windows, there is an undocumented flag called ThreadHideFromDebugger. When you apply this flag to a thread, the kernel stops sending debug events (like breakpoints or exceptions) from that thread to any attached debugger.
The result: If an analyst tries to set a breakpoint in a "hidden" thread, the program will simply crash the moment it hits the breakpoint, because the debugger never "hears" the signal to pause.
2.Since this function is undocumented, we can't just call it normally. We have to find it inside ntdll.dll while the program is running.3. Why this is the "Final Boss" of your library:
3. Why this is the "Final Boss" of your library:
   Invisible Breakpoints: If the analyst sets a breakpoint in a hidden thread, the debugger won't catch it. The application will just terminate.
    Anti-Dumping: It makes it much harder for tools to "dump" the process memory while it's running.
    Silent Fail: It doesn't return a bool. It just changes the "rules" of the operating system for that specific thread



#include <Windows.h>

// Define the function signature for the undocumented NTAPI
typedef NTSTATUS (NTAPI* pNtSetInformationThread)(
    HANDLE ThreadHandle,
    UINT ThreadInformationClass,
    PVOID ThreadInformation,
    ULONG ThreadInformationLength
);

void HideThread() {
    // 1. Get the handle to ntdll.dll
    HMODULE hNtdll = GetModuleHandleA("ntdll.dll");
    if (hNtdll) {
        // 2. Find the address of NtSetInformationThread
        auto NtSetInfoThread = (pNtSetInformationThread)GetProcAddress(hNtdll, "NtSetInformationThread");

        if (NtSetInfoThread) {
            // 3. 0x11 is the 'ThreadHideFromDebugger' constant
            // We apply it to the current thread (GetCurrentThread())
            NtSetInfoThread(GetCurrentThread(), 0x11, NULL, 0);
        }
    }
}

 


Here is the breakdown of why this is so powerful:
1. The Target: ntdll.dll
cpp

HMODULE hNtdll = GetModuleHandleA("ntdll.dll");

Use code with caution.

    What it is: ntdll.dll is the lowest-level library in Windows user-mode. It is the gateway to the Kernel.
    Why do this? The function we need (NtSetInformationThread) is "undocumented." This means it’s not in the standard Windows headers. We have to manually find its address in memory while the program is running.

2. Finding the Secret Function
cpp

auto NtSetInfoThread = (pNtSetInformationThread)GetProcAddress(hNtdll, "NtSetInformationThread");

Use code with caution.

    What it is: GetProcAddress acts like a phone book. You are asking: "Where exactly inside ntdll.dll is the code for NtSetInformationThread?"
    Why do this? Since we can't link to it at compile time, we "link" to it at runtime. This also makes your code harder to analyze statically because the function name isn't in your program's "Import Table."

3. The "Kill Switch": 0x11
cpp

NtSetInfoThread(GetCurrentThread(), 0x11, NULL, 0);

Use code with caution.

    The Magic Number: 0x11 (decimal 17) is the ID for ThreadHideFromDebugger.
    The Effect: You are telling Windows: "From now on, do not send any debug messages (breakpoints, exceptions, stepping) for this specific thread to the debugger."
    The Result: If an analyst sets a breakpoint in this thread, the CPU will hit the breakpoint, but the debugger will never be notified. Since the debugger doesn't "handle" the break, the program simply crashes. To the analyst, it looks like their debugger is broken.

4. Why this is the "Nuclear Option"

    Un-traceable: Once this runs, the debugger is essentially "blind" to this thread.
    No "True/False": Notice this function doesn't return a value to check. It just changes the environment.
    Bypass: The only way for an analyst to stop this is to find your call to NtSetInformationThread and "NOP" (delete) it in memory before it executes.
